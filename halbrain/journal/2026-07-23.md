**What worked + Evidence:**
- **Finding 1 — Open redirect (`apps/web/src/app/api/auth/callback/route.ts:41`, verified conf 8/10):** `redirect` query param was raw-concatenated `${APP_URL}${redirect}`. Since `APP_URL` (`https://helloagainlinks.com`) has NO trailing slash, `redirect=.evil.com/phish` → `https://helloagainlinks.com.evil.com/phish` (attacker host); `@evil.com` variant → userinfo escape. `login/route.ts:15` forwarded the param verbatim into the OAuth `redirectTo`, so it survived the round-trip. **Fix:** new `apps/web/src/lib/safe-redirect.ts` → `safeInternalPath(raw, fallback)` accepts only same-origin relative paths (must start with single `/`, reject `//`, backslash, control chars incl. CR/LF). Wired into the callback sink and login forward (defense-in-depth). Cookies were already httpOnly+SameSite=Lax so impact was phishing not token theft → was Medium.
- **Findings 2+3 — SECURITY DEFINER RPC IDOR (`supabase/migrations/002` search_bookmarks + `006` get_folders_with_counts, verified conf 7/10 & 6/10):** both are `SECURITY DEFINER` (bypass RLS) and filter by a **caller-supplied `p_user_id`** instead of `auth.uid()`. Postgres grants EXECUTE to PUBLIC by default (no REVOKE) → reachable with the public anon key: knowing a victim's user UUID leaks their bookmark-id/search-count or folder metadata. **Fix = new forward migration `010_fix_definer_rpc_authz.sql`** (historical migrations immutable/already applied). Callers verified first: BOTH `search_bookmarks` callers use `ctx.serviceClient` (bookmarks/search route + grok `buildBookmarkContext` via conversations/[id]/messages:111) → locked EXECUTE to `service_role` + REVOKE from PUBLIC/anon/authenticated + pinned `search_path`. `get_folders_with_counts` is called with `ctx.userClient` (authenticated role, folders/route.ts:33) so it MUST stay executable by `authenticated` → instead bound identity inside the body: `AND (auth.uid() IS NULL OR f.user_id = auth.uid())` + REVOKE from PUBLIC/anon (closes the auth.uid()-IS-NULL escape for unauth callers; NULL branch left only for trusted service_role reuse).
- **Resolved a disagreement between two verifier agents:** `get_related_bookmarks` (007) is `LANGUAGE sql` **without** SECURITY DEFINER (runs as invoker → RLS still constrains it) → NOT vulnerable, left unchanged. Confirmed by reading 007 directly (lines 17-18).
